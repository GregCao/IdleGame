## 1. 真机测试清单

### 1.1 Unity WebGL 构建

| 检查项 | 状态 | 说明 |
|--------|------|------|
| Build Settings → WebGL 平台选中 | ☐ | 确认 Platform = WebGL |
| Player Settings → Publishing Settings 勾选 | ☐ | Compression Format = Gzip/Brotli |
| Player Settings → Resolution 设置 | ☐ | Default Canvas Width/Height 适配 750px |
| Player Settings → Run In Background = true | ☐ | 防止切后台断连 |
| IL2CPP 编译后无报错 | ☐ | 确保代码无裁剪错误 |
| Build 大小 < 15MB（首包） | ☐ | 微信首包限制 2MB，需做分包 |
| 分包策略制定 | ☐ | 主包 + 资源包分离 |
| 内存占用 < 256MB | ☐ | WebGL 内存限制 |
| 启动时间 < 5s（弱网 < 10s） | ☐ | 用户留存关键指标 |

### 1.5 常见问题排查

| 问题现象 | 可能原因 | 解决方案 |
|----------|----------|----------|
| 微信提示"游戏加载失败" | 安全域名未配置 | 登录公众平台 → 开发管理 → 服务器域名 → 添加 request/download 合法域名 |
| 白屏时间过长 | 首包过大/未做分包 | 压缩资源，制作分包加载 |
| 战斗数字不显示 | Canvas 层级被遮挡 | 检查 CSS z-index 设置 |
| 激励视频无法播放 | ad unit 未配置 | 微信后台广告位申请，配置正确广告位 ID |
| 支付失败 | 微信支付未开通 | 商户平台开通 Native/JSAPI 支付 |
| 分享回调无响应 | 分享参数缺失 | 配置 `wx.showShareMenu` + 定制分享图 |
| 切后台后游戏重置 | 存档机制问题 | 检查 PlayerPrefs 保存时机 |
| iOS 上音效卡顿 | AudioSource 未做池化 | 实现对象池复用 AudioSource |
| Android 上偶发闪退 | 内存泄漏 | 使用 DevTools 检测 JS Heap 增长 |

## 2. 微信审核清单

### 2.1 资质要求

| 主体类型 | 所需资质 | 适用场景 |
|----------|----------|----------|
| 个人开发者 | 身份证 + 微信实名认证 | 小规模运营，IAP 收入有限 |
| 企业开发者 | 营业执照 + 对公账户验证 | 大规模运营，需开通微信支付 |
| 个体工商户 | 营业执照 + 法人身份证 | 可开通部分支付功能 |

> **注意**：放置类游戏若含充值功能（IAP），需企业主体并开通微信支付。

### 2.2 材料准备

| 材料 | 说明 | 状态 |
|------|------|------|
| 软件著作权（软著） | 必须，含游戏名称"放置江湖" | ☐ |
| 游戏版号 | 国产游戏需版号（个人可申请但流程长） | ☐ |
| ICP 备案 | 域名需有 ICP 备案 | ☐ |
| 隐私政策文档 | 独立页面，注明数据收集范围 | ☐ |
| 用户协议文档 | 独立页面，含退款条款 | ☐ |
| 游戏图标 | 尺寸 108×108px，PNG 格式 | ☐ |
| 游戏封面图 | 16:9 竖版，2MB 以内 | ☐ |
| 游戏截图 | 4-5 张，展示核心玩法 | ☐ |
| 激励视频广告 ID | 微信后台申请 | ☐ |
| 微信支付商户号 | 商户平台申请 | ☐ |
| 最低版本说明 | iOS ≥ 9.0，Android ≥ 6.0 | ☐ |

### 2.3 提交审核流程

```
1. 登录微信公众平台 (https://mp.weixin.qq.com)
2. 进入"版本管理" → "上传代码"
3. 填写版本号和备注
4. 提交审核 → 填写游戏类目申请
   ├── 类目选择：休闲游戏
   ├── 所需资质：软著/版号（见2.2）
   └── 内容真实性声明
5. 等待审核（通常 3-7 个工作日）
6. 审核通过 → 选择"发布"或"提交发布"
7. 发布后 1-2 小时内全量上线
```

| 步骤 | 操作 | 状态 |
|------|------|------|
| 代码上传 | 开发者工具 → 上传 | ☐ |
| 类目选择 | 休闲游戏 | ☐ |
| 资质上传 | 软著 + 备案 | ☐ |
| 版本备注 | 如"v1.0.0 首版上线" | ☐ |
| 提交审核 | 点击提交 | ☐ |
| 审核通过发布 | 确认发布 | ☐ |

### 2.4 审核常见驳回原因

| 驳回原因 | 解决方案 |
|----------|----------|
| 缺少软件著作权 | 提前申请软著（30-50 个工作日），或使用版号 |
| 类目选择错误 | 确认选择"休闲游戏"而非其他类目 |
| 隐私政策不完善 | 补充数据收集目的、第三方 SDK 声明 |
| 诱导分享 | 移除强制分享机制 |
| 诱导充值 | 去除"不充值无法继续"等文案 |
| 虚拟货币比例不透明 | 明确标注汇率和充值比例 |
| 广告标识不清晰 | 激励视频需标注"广告"字样 |
| 出现外链 | 移除所有外部链接 |
| 最低版本过高 | iOS 降至 9.0，Android 降至 5.0 |
| 截图与实际不符 | 使用实际游戏截图 |

## 4. 后续运营

### 4.1 版本更新流程

```
需求评审 → 开发 → 内测 → 提交审核 → 审核通过 → 全量发布
```

| 阶段 | 步骤 | 周期 | 状态 |
|------|------|------|------|
| 需求评审 | PRD 评审 + 排期 | 1-2 天 | ☐ |
| 开发阶段 | 功能开发 + 自测 | 3-7 天 | ☐ |
| 内测版本 | 邀请 50-100 名用户测试 | 2-3 天 | ☐ |
| 修复阶段 | BUG 修复 + 回归测试 | 1-2 天 | ☐ |
| 提交审核 | 开发者工具上传 → 提交审核 | 1 天 | ☐ |
| 审核等待 | 等待微信审核 | 3-7 天 | ☐ |
| 全量发布 | 审核通过后发布 | 1 天 | ☐ |

| 版本命名规则 | 格式 |
|--------------|------|
| 主版本 | 大版本功能迭代（如 v2.0.0） |
| 次版本 | 功能新增（如 v1.1.0） |
| 修订版本 | BUG 修复（如 v1.0.1） |
